# Collaboration Retrospective Cadence

## 目标

把“每 n 轮对话做一次回顾和反思”固化为项目协作规则，让开发过程能持续改进，而不是依赖临时记忆。

## 背景

当前项目已经形成多个长期协作约束：

- 每个模块开发前先写模块文档。
- 低决策任务可以下发 agent，但主 agent 负责审计和集成。
- 每次变更需要测试和构建证据。

缺口是缺少固定复盘节奏。没有节奏时，复盘通常只会在出错后发生；这会导致问题重复出现，例如 agent 数量膨胀、计划漂移、上下文负担增加、模块边界不清晰。

## 契约

- 默认每 10 个用户-助手交互轮次，或每完成一个有意义模块后，做一次短复盘。
- 若正在执行关键验证、修复失败构建或处理用户明确的紧急请求，可以先完成当前关键步骤，再补复盘。
- 复盘输出必须短，只回答四件事：
  - 本阶段做对了什么。
  - 本阶段暴露了什么协作或工程风险。
  - 下一阶段应调整什么规则或流程。
  - 是否需要写入项目文档或记忆。
- 复盘不得替代实现、测试或用户要求的交付物。
- 只有用户明确要求保存为长期记忆时，才写入本地 memory；否则只更新项目文档或当前计划。

## 边界

- 本模块只定义协作流程，不新增运行时代码。
- 不强制精确计数历史所有对话轮次，采用阶段性触发。
- 不在紧急修复中插入长篇总结，避免打断主路径。

## 实施结果

- 总计划新增 `Collaboration retrospective cadence` 协作契约。
- 默认节奏设为每 10 个用户-助手交互轮次，或每完成一个有意义模块后。
- 复盘输出固定为四项：做对了什么、暴露什么风险、下一阶段调整什么、是否需要写入项目文档或记忆。
- 明确复盘不得替代实现、测试或用户要求的交付物。
